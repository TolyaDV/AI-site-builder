# /new-site — створити новий сайт

## Крок 0: Перевірка середовища
- Переконайся що в директорії немає існуючого Next.js проєкту (`package.json` з залежністю `next`). Якщо є — СТОП, попереди користувача.
- Переконайся що встановлено Node 20+ і git.
- Якщо є блокер — зупинись і скажи користувачу що виправити.

## Крок 1: Бриф
Задавай питання ПО ОДНОМУ, чекай відповіді. Після кожного блоку — підсумовуй і рухайся далі.

1. **Бізнес**: ніша, що продає/робить, ціль сайту (ліди / продажі / імідж / інфо).
2. **Клієнт**: назва проєкту, короткий опис у 1-2 реченнях.
3. **Аудиторія і тон**: хто читач, який тон комунікації (professional / playful / luxury / technical).
4. **Мова**: мова UI і контенту (uk / en / інша).
5. **Тип сайту**: односторінковий (landing) чи багатосторінковий (multipage)?
6. **Структура**:
   - Якщо landing — які секції потрібні (hero, features, pricing, testimonials, CTA тощо). Оптимальний порядок запропонуєш сам на етапі дизайну.
   - Якщо multipage — які сторінки і що на них.
   - Якщо користувач не знає — запропонуй стандартний набір під нішу і тип, погодь.
7. **Дизайн-референси**: чи є готовий макет / референс-сайти / логотип / шрифти / кольори. Якщо чогось немає — запитай: підібрати самому чи додати пізніше. Усе «пізніше» — записати в `brief.md` як розділ `## TODO`.
8. **Контакти і legal**: телефон, email, адреса, соцмережі, потреба в privacy/cookies. Якщо немає зараз — у `## TODO`.
9. **Деплой**: куди плануємо публікувати — `Vercel` / `Netlify` / `Cloudflare Pages` / `інше` / `ще не знаю`. Відповідь записується в `brief.md` секцією `## Deploy`. Якщо «не знаю» — у `## TODO`, `/deploy` уточнить при першому виклику.

Після всіх відповідей — збери `brief.md` і покажи користувачу. Чекай підтвердження ("ок" / "змінити X"). Не рухайся далі без підтвердження.

## Крок 2: Дизайн-система
На основі `brief.md` скопіюй `.claude/templates/design-system.md` у `./design-system.md` і заповни:
- Палітра (primary, accent, neutral, semantic) — під нішу і тон.
- Типографіка (heading/body шрифти через `next/font`, масштаб).
- Spacing scale, border-radius, shadows.
- Стилі кнопок, карток, секцій, форм.

Покажи користувачу. Чекай підтвердження.

## Крок 3: Ініціалізація проєкту
Виконай послідовно:
1. `npx create-next-app@latest . --typescript --tailwind --app --src-dir --import-alias "@/*" --eslint --yes`
2. `npx shadcn@latest init` (base color — узгодь з палітрою з design-system.md)
3. `npm i next-sitemap`
4. Налаштуй `tailwind.config.ts` з токенами з `design-system.md` (кольори, шрифти, radius, spacing).
5. Скопіюй `.claude/references/globals.reference.css` у `src/app/globals.css` і заміни OKLCH-значення на актуальні з `design-system.md`. Не пиши `globals.css` з нуля.
6. Створи структуру: `src/components/ui/`, `src/components/sections/`, `src/lib/`.
7. `git init && git add -A && git commit -m "chore: initial scaffold"`

## Крок 4: Створення сторінок
- **Layout і metadata**: `src/app/layout.tsx` з `next/font`, базовим `metadata` (title template, OG-дефолти), семантичним HTML.
- **Header і Footer** у `src/components/sections/` — використовуй реальні контакти з брифу. Якщо контактів немає — вживай фіксовану плашку `{{ TODO: контакт }}` і додай у `brief.md` відповідний TODO. НЕ Lorem ipsum.
- **Контент**:
  - Landing → одна сторінка `src/app/page.tsx` із секціями з брифу у заданому порядку.
  - Multipage → окрема `src/app/<route>/page.tsx` для кожної сторінки з брифу.
  - Для кожної: унікальні `metadata` (title/description), реальний контент, OG-картинка.
- **Production-checklist**: застосуй усі пункти з `.claude/references/production-checklist.md`, релевантні до створення (метадані, OG, favicon, 404, a11y, аналітика, форми). Legal і чисту контент-перевірку можна відкласти до `/deploy`, але все технічне — зараз.

## Крок 5: Favicon і OG-image
- Створи `src/app/icon.tsx` або `public/favicon.ico` — брендована іконка (використай основний колір палітри, першу літеру назви або простий символ з ніші).
- Створи `src/app/apple-icon.tsx` (180×180).
- Створи `src/app/opengraph-image.tsx` (1200×630) — динамічний OG-image через Next.js API, з назвою проєкту і брендовими кольорами.
- Альтернативно: якщо користувач надав логотип у брифі — використай його.

## Крок 6: Перевірка
Запусти послідовно. Якщо падає — ВИПРАВ і повтори. Не переходь далі з червоним:
1. `npx tsc --noEmit` → 0 помилок.
2. `npm run lint` → 0 помилок.
3. `npm run build` → успіх.
4. Запусти `npm run dev` у фоні та повідом користувача: «Сайт працює на http://localhost:3000. Перевір візуально всі сторінки в браузері і скажи що правити». Якщо підключено Playwright MCP — зроби скриншоти головних сторінок і додай у повідомлення.

## Крок 7: Завершення
- Оновити `CHANGELOG.md`: дата, «Initial site scaffold + pages X, Y, Z», список TODO з брифу.
- `git add -A && git commit -m "feat: initial pages and design system"`
- Повідом користувача: сайт готовий локально, список TODO з брифу, наступні кроки.
