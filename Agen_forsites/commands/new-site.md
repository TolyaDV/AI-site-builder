# /new-site — створити новий сайт

## Крок 0: Перевірка середовища
- Переконайся що в директорії немає існуючого Next.js проєкту (`package.json` з залежністю `next`). Якщо є — СТОП, попереди користувача.
- Переконайся що встановлено Node 20+ і git.
- Якщо є блокер — зупинись і скажи користувачу що виправити.

## Крок 1: Бриф
Починай з Кроку 0 (вільний опис). Далі задавай питання ПО ОДНОМУ блоку, чекай відповіді, після кожного блоку — підсумовуй і рухайся далі. Якщо користувач уже розповів щось у Кроці 0 — не переспитуй те саме, тільки дошукай прогалини.

0. **Вільний опис**. Постав дослівно:
   > «Розкажи про свій проєкт як хочеш — що це, для кого, що має бути на сайті, які відчуття має викликати. Без структури, просто думки і побажання.»

   Коли користувач відповів — зроби короткий summary (3-5 рядків: що ти зрозумів про бізнес, аудиторію, настрій, функції) і запитай «правильно?». Після підтвердження переходь до наступних блоків, витягуючи тільки те, чого не вистачає.

1. **Бізнес**:
   - Назва проєкту
   - Ніша і що саме продає/робить (одне питання)
   - Ціль сайту: ліди / продажі / імідж / інфо

2. **Аудиторія і тон**:
   - Портрет ЦА
   - Тон: professional / playful / luxury / technical
   - Мова UI і контенту: uk / en / інша

3. **Конкуренти**:
   - 1-2 сайти конкурентів
   - Чим хочемо відрізнятись (позиціонування, не візуал — візуальні референси — наступний крок)

4. **Логотип** — якщо є у клієнта:
   - Файл / URL / нема

   Якщо нема — записати `нема` (не `TODO`), скіл `design-system` на Кроці 3 підбере direction.

   **Примітка**: все інше з дизайну (шрифти, палітра, готовий макет, inspiration-референси, настрій, стиль) НЕ питаємо тут — їх питає скіл `design-system` на Кроці 3 `/new-site`, коли юзер уже у «дизайн-моді». Тут — тільки логотип як якір бренду.

5. **Тип сайту**: Landing чи Multipage.
   - Конкретний перелік секцій (для landing) або сторінок (для multipage) НЕ питай — агент сформує його автономно з брифу на етапі створення сторінок.
   - Якщо користувач не знає навіть тип — запропонуй під нішу і ціль, погодь.

6. **Функції сайту**: які інтерактивні елементи потрібні. Дай явний перелік категорій у питанні:
   контактна форма / форма заявки / booking-календар / калькулятор / онлайн-чат / галерея / відгуки / e-commerce (кошик, оплата) / auth (особистий кабінет) / мультимовність / CRM-email інтеграції / cookie-banner / інше.

   Якщо не знає — запиши `## TODO: features (додати по ходу)` і не блокуй процес. Функції впливатимуть на стек — фіксуй кожну, навіть «можливо потрібно».

7. **Контент**:
   - Хто пише тексти: клієнт / агент генерує / копірайтер
   - Фото / відео: готові у клієнта / stock / генерувати / немає

   Якщо нічого немає — `## TODO: контент`. Важливо знати заздалегідь чи генерувати тексти, а не тільки вставляти `{{ TODO }}`-маркери.

8. **Контакти і legal**:
   - Телефон, email, адреса, соцмережі (компанії, НЕ власника — див. CLAUDE.md)
   - Privacy / cookies сторінки — потрібні?

   Якщо немає — у `## TODO`.

9. **Технічне**:
   - Деплой: Vercel / Netlify / Cloudflare Pages / інше / не знаю
   - Домен: вже є / треба купити / піддомен
   - Аналітика: GA / Plausible / нічого

   Відповідь про деплой записується в `brief.md` секцією `## Deploy`. Якщо «не знаю» — у `## TODO`, `/deploy` уточнить при першому виклику.

Після всіх блоків — збери `brief.md` (з секціями `## Вільний опис`, `## Бізнес`, ..., `## TODO`, `## Deploy`) і покажи користувачу. Чекай підтвердження ("ок" / "змінити X"). Не рухайся далі без підтвердження.

## Крок 2: Структура сайту
Запусти скіл `site-structure` — він читає `brief.md` і створює `structure.md` (тип landing/multipage + список секцій/сторінок з intent на кожну + дизайн-директива у блок `## Design Direction` брифу).

Після того як скіл показав структуру — чекай підтвердження юзера. Далі.

## Крок 3: Дизайн-система
Запусти скіл `design-system` у init-режимі. Він сам:
- Запитає чи є юзерські референси (URL / скріншот / код / опис).
- Якщо є — проаналізує, збереже у `./design-refs/`, витягне патерни.
- Якщо немає — запропонує 2-3 варіанти палітри під archetype з брифу.
- Створить `./design-system.md` з палітрою (light+dark), типографікою, radius, motion, контраст-таблицею.
- Синхронізацію з кодом НЕ виконуватиме (проєкт ще не існує) — тільки пропише `design-system.md`.

Після підтвердження дизайн-системи — рухайся далі.

## Крок 4: Ініціалізація проєкту
Виконай послідовно:
1. `npx create-next-app@latest . --typescript --tailwind --app --src-dir --import-alias "@/*" --eslint --yes`
2. `npx shadcn@latest init` (base color — узгодь з палітрою з `design-system.md`)
3. `npm i next-sitemap`
4. Створи структуру: `src/components/ui/`, `src/components/sections/`, `src/lib/`.
5. **Повторно запусти скіл `design-system`** — тепер проєкт існує, скіл синхронізує: `tailwind.config.ts`, `src/app/globals.css` (копіює з `.claude/references/globals.reference.css` + підставляє актуальні OKLCH), `src/app/layout.tsx` (next/font imports).
6. `git init && git add -A && git commit -m "chore: initial scaffold + design-system"`

## Крок 5: Створення сторінок
Джерело структури — `structure.md` (створений скілом у Кроці 2). Для кожної секції/сторінки у ньому є intent (коротке речення про призначення) — використовуй як brief для написання контенту.

- **Layout і metadata**: `src/app/layout.tsx` з `next/font`, базовим `metadata` (title template, OG-дефолти), семантичним HTML.
- **Header і Footer** у `src/components/sections/` — використовуй реальні контакти з брифу. Якщо контактів немає — вживай фіксовану плашку `{{ TODO: контакт }}` і додай у `brief.md` відповідний TODO. НЕ Lorem ipsum.
- **Контент**:
  - Landing → одна сторінка `src/app/page.tsx` із секціями з `structure.md` у заданому порядку.
  - Multipage → окрема `src/app/<route>/page.tsx` для кожної сторінки з `structure.md`.
  - Для кожної: унікальні `metadata` (title/description), реальний контент за intent-описом, OG-картинка.
- **Production-checklist**: застосуй усі пункти з `.claude/references/production-checklist.md`, релевантні до створення (метадані, OG, favicon, 404, a11y, аналітика, форми). Legal і чисту контент-перевірку можна відкласти до `/deploy`, але все технічне — зараз.

## Крок 6: Favicon і OG-image
- Створи `src/app/icon.tsx` або `public/favicon.ico` — брендована іконка (використай основний колір палітри, першу літеру назви або простий символ з ніші).
- Створи `src/app/apple-icon.tsx` (180×180).
- Створи `src/app/opengraph-image.tsx` (1200×630) — динамічний OG-image через Next.js API, з назвою проєкту і брендовими кольорами.
- Альтернативно: якщо користувач надав логотип у брифі — використай його.

## Крок 7: Перевірка
Запусти послідовно. Якщо падає — ВИПРАВ і повтори. Не переходь далі з червоним:
1. `npx tsc --noEmit` → 0 помилок.
2. `npm run lint` → 0 помилок.
3. `npm run build` → успіх.
4. Запусти `npm run dev` у фоні та повідом користувача: «Сайт працює на http://localhost:3000. Перевір візуально всі сторінки в браузері і скажи що правити». Якщо підключено Playwright MCP — зроби скриншоти головних сторінок і додай у повідомлення.

## Крок 8: Завершення
- Оновити `CHANGELOG.md`: дата, «Initial site scaffold + pages X, Y, Z», список TODO з брифу.
- `git add -A && git commit -m "feat: initial pages and design system"`
- Повідом користувача: сайт готовий локально, список TODO з брифу, наступні кроки.
